# Phase 1 Kickoff Brief — for a Fresh Opus Chat

You are picking up the C2 (Command & Control) infrastructure project for the ECS Simulation Engine at the start of Phase 1. Phase 0 — the build-out of the orchestration factory — is complete. This document is your bootstrap context. Read it once, then read the reference docs it points at, then begin.

---

## What this project is

A headless C# / .NET 8 entity-component-system simulation engine being grown into a 2.5D top-down management sim of an office — in the lineage of The Sims, Rimworld, Prison Architect, with the charm-density of *Grossology* and the running-gag economy of *The Office*. The player is a ghost-camera who watches and influences the lives of office workers. Drives, relationships, memories, schedules, romance, office politics. Lived-in, not sterile.

The engine and visualization runtime are 100% offline. The Anthropic API is a *design-time* tool: Sonnets generate code and content at build time, Haikus validate balance at build time. The runtime game makes zero LLM calls.

## What Phase 0 built

A 13-project .NET 8 solution. Four original engine projects (`APIFramework`, `APIFramework.Tests`, `ECSCli`, `ECSVisualizer`) and nine new `Warden.*` projects implementing:

- Strict JSON schemas for cross-tier handshake (`Warden.Contracts`).
- Telemetry projection from internal `SimulationSnapshot` to wire-format `WorldStateDto` (`Warden.Telemetry`).
- A thin Anthropic Messages + Batches HTTP client (`Warden.Anthropic`).
- A full orchestrator binary (`Warden.Orchestrator`) with: prompt cache manager (4-slab model), batch scheduler (deduplicated, ≤25 scenarios), cost ledger (JSONL + budget guard), concurrency controller (`Task.WhenAll` + `SemaphoreSlim(5)`), chain-of-thought persistence, fail-closed escalator, and report aggregator.

End-to-end smoke runs (mock-Anthropic and real-API under $1) have been validated. Every work packet has a completion note in `docs/c2-infrastructure/work-packets/_completed/`. SRD §6 acceptance criteria are met.

## The architectural axioms

These are recorded in `docs/c2-infrastructure/00-SRD.md` §8 and they do not change. Memorize them; every Phase 1+ packet must respect them.

1. **Anthropic API is design-time only.** Runtime engine has zero LLM dependency. Any packet that proposes a runtime LLM call is rejected on architectural grounds.
2. **Save/load reuses `WorldStateDto`.** No second save format. Replay (deterministic test) and save/load (player's world) are different concepts on the same storage shape.
3. **Memory model: per-pair primary, global thin.** NPC↔NPC memories live on the relationship entity. Office-wide events ("Kevin's chili") live in a thin global chronicle that per-NPC memories reference by id.
4. **Charm is curated, not simulated.** Dust, stacked boxes, the parking-lot "no fighting" sign — placed at world start, not generated by entropy. Some events do persist (spilled coffee → stain that stays), but they flow through the v0.3 chronicle channel, not via dust accumulation models.
5. **Social state is a first-class component family.** Drives, relationships, memories, schedules will be the largest live component family in the engine, not an afterthought.
6. **Visual target is 2.5D top-down management sim with pixel-art aesthetic.** The headless engine doesn't render, but it must expose room-membership and lighting state for AI-tier reasoning about *places*, not just float coordinates.

## Phase 1 scope

The factory exists. Phase 1 starts using it to produce content in a deliberate sequence. The order is significant — don't skip ahead.

### Phase 1.0 — Schema v0.2 (Social Pillar)

**Goal:** Extend the schema, contract, and telemetry layers to express social state — drives, relationships, memories — without breaking v0.1 consumers.

This is the foundational move. Everything in Phase 2+ rides on it. Drafted as 4 small packets:

- **WP-1.0.A** — schema additions (`world-state` v0.2 with optional `social` slots, top-level `relationships[]` and `memoryEvents[]`). Update validators. Round-trip tests.
- **WP-1.0.B** — `Warden.Contracts.Handshake.Social` DTOs (records). Big Five personality struct. Drive vector with eight social drives (belonging, status, affection, irritation, attraction, trust, suspicion, loneliness). Memory and relationship records.
- **WP-1.0.C** — engine social-component family (`APIFramework/Components/Social/`) plus a placeholder `SocialDriveDecaySystem` that does nothing useful yet but proves the wiring. The `TelemetryProjector` learns to project social state into v0.2 telemetry.
- **WP-1.0.D** — `SCHEMA-ROADMAP.md` updates and a v0.1↔v0.2 migration test.

Estimated cost: ~$1.50 across Sonnets, ~$0 Haikus (no balance work yet).

### Phase 1.1 — Narrative Telemetry Channel

**Goal:** Engine emits cheap string-templated event summaries — *"Frank entered the breakroom; Donna left the bathroom (occupancy 14m); coffee pot is empty (4th time today)"* — that Haikus consume to validate behavior without rendering.

- **WP-1.1.A** — `INarrativeDigestEmitter` interface in `APIFramework`, plus a `RingBufferDigest` capping at 256 events per frame.
- **WP-1.1.B** — Telemetry DTO learns a `narrativeDigest[]` field (still v0.2 minor — additive). Engine systems emit named events through the digest.
- **WP-1.1.C** — `ECSCli ai snapshot` and `ai stream` include the digest in their output.

### Phase 1.2 — World-Bible Bootstrap

**Goal:** Translate Talon's `world-bible.md` into a data-driven world definition the engine bootstraps from at game start.

- **WP-1.2.A** — `world-definition.schema.json` (v0.2 minor) for declaring rooms, named locations, baseline state.
- **WP-1.2.B** — `Warden.WorldBuilder` project that reads a world-definition JSON and produces an entity-placement script for `SimulationBootstrapper`.
- **WP-1.2.C** — Author the first JSON form of the world from `world-bible.md`. Sonnets translate the prose into the schema.

### Phase 1.3 — Cast-Bible Bootstrap

Same pattern as 1.2, but for the cast. Produces a `character-catalog.json` (v0.4 schema preview) and engine-side instantiation.

### Phase 1.4 — Persistent Chronicle Layer (v0.3 Schema Bump)

Wire the chronicle into telemetry. First persistent-event recipe: spilled coffee → stain entity that persists across save/load. The Office-style "events stay happened" mechanic becomes real.

### Phase 1.5 — Aesthetic & Lighting State (v0.5/v0.6 Schema Bumps)

Room overlay and per-tile illumination state. The aesthetic-bible's lighting → behavior mapping becomes engine state (a flickering fluorescent raises the affected NPCs' anxiety drive).

## Operational reality

You (Opus) author specs; you don't execute code. Your output is `OpusSpecPacket` JSON (one per Sonnet) plus the WP markdown that frames it. Talon hand-carries the specs to the orchestrator; the orchestrator dispatches Sonnets in parallel up to `SemaphoreSlim(5)`.

Each packet you write should follow the format established by `docs/c2-infrastructure/work-packets/WP-NN-*.md`:

- Header (tier, depends-on, timebox, budget).
- Goal (one paragraph — the "why").
- Reference files (read these, do not read others).
- Non-goals (hard forbidden — keeps Sonnets in scope).
- Deliverables (table of code files with one-line descriptions).
- Acceptance tests (table of ID, assertion, verification method).

The non-goals section is the highest-leverage way to keep token costs low. Every "do not" saves a Sonnet a wrong turn.
