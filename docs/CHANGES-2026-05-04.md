**Files:** `APIFramework.Tests/Determinism/FullSimulationDeterminismTests.cs` (5 tests).

**What:** Existing determinism tests cover narrow slices (ComponentStore, mutation, lockout, slip-and-fall). NONE verifies that the full SimulationBootstrapper — with all 50+ systems firing every tick — produces deterministic outcomes across two independent runs.

This packet adds 5 full-sim determinism tests:

- 10 NPCs × 500 ticks (multi-entity state-space)
- 30 NPCs × 200 ticks (canonical playtest scale; the headline check)
- Different-seed divergence (sanity guard against tautological pass)

Each test runs two SimulationBootstrappers with identical seeds, ticks them in lockstep, then compares a state digest covering every human's metabolism, energy, position, drives, and bladder fill. Failures dump the first divergent line with surrounding context for forensic diff-friendly output.

**Why this matters NOW (specifically for Phase 4.2.x):**
- WP-4.2.0 (zone substrate) needs each zone to be deterministic in isolation.
- **WP-4.2.2 (LoD) is most at risk** — coarsening a system to run every Nth tick changes the order of operations relative to other systems. These tests catch the baseline NOW; LoD-introduced regressions show up as test failures rather than as "the same scenario plays out differently each time."
- Save/load fidelity (already shipped Wave 4) implicitly depends on determinism — these tests are also the regression guard for that.

## 6. Performance regression guards — merged `7e2f79b`

**Files:** `APIFramework.Tests/Performance/PerformanceRegressionGuardTests.cs` (3 tests).

**What:** Hard-failing assertive guards complement the regenerable fixture. The fixture diff is process — humans review during PR. This is automation — failing build catches the "we accidentally introduced a 60-second tick" class of regression with no human in the loop.

Three guards:
- 30-NPC tick median **< 5.0 ms** (~14× headroom over current ~0.35 ms baseline)
- 100-NPC tick median **< 16.0 ms** (one full 60-FPS frame budget)
- Scaling: 100-NPC tick must NOT exceed 30-NPC tick by **> 10×** (catches accidental O(N²) growth in any system)

Thresholds deliberately generous so:
- Real regressions trip the guards.
- Normal CI noise (thermal throttling, concurrent test load) doesn't.
- Subtle 25–50% regressions still flow through fixture-diff review.

## 8. Allocation hot-path investigation — merged `d54c12c` (+ flake fix `b18bab0`)

**Files:** `APIFramework.Tests/Performance/AllocationHotPathInvestigationTests.cs` + checked-in fixture `APIFramework.Tests/Performance/PerformanceFixtures/allocation-hot-paths.md`.

**What:** Diagnoses the per-system allocation floor surfaced by `system-baselines.md`. Probes minimal ISystem implementations (NoOp / QueryOnly / QueryEnumerate / QueryAndGet) to identify where the bytes-per-call floor comes from.

**Findings (humanCount=30):**

| Probe | Update body | bytes / call |
|:---|:---|---:|
| NoOpSystem | (empty) | **0** |
| QueryOnlySystem | em.Query<T>().Count() | 4 |
| QueryEnumerateSystem | foreach over Query | **44** |
| QueryAndGetSystem | foreach + Get<T> per entity | 44 |

**Key insights:**
- NoOpSystem allocates 0 bytes — confirms the measurement harness is leak-free; the per-system numbers in `system-baselines.md` are reliable.
- The 44-byte cost is the `foreach` enumerator boxing. **Constant regardless of NPC count** (1, 10, 30, 60, 100 all show 44 bytes) — meaning it doesn't worsen at scale.
- `Get<T>()` adds **0 bytes** — validates that the WP-3.0.5 ComponentStore<T> refactor achieved its zero-allocation goal.

**Action items recorded in fixture for any future allocation-reduction packet:**
1. Eliminating the 44-byte foreach overhead via a struct-enumerator IEntityQuery would save ~44 bytes × 50 systems × 60 FPS = 132 KB/sec across the engine. Meaningful but not critical at current scales.
2. The remaining ~270 bytes/call in real systems comes from system-internal code (multiple foreach loops, string ops, etc.), not framework overhead.

**Flake fix `b18bab0`:** The sanity guard `Assert.True(noop < 1024)` failed under full-suite parallel test execution because `GC.GetTotalAllocatedBytes` measures process-wide allocations. Loosened to 100 KB to tolerate concurrent test noise while still catching real harness leaks. Lesson learned in code comment + this doc.

**Tests:** 1 new.
