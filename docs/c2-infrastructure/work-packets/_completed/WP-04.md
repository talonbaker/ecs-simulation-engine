## Determinism fixes applied

The following nondeterminism sources were found and fixed (per WP-04 instruction to fix the source, not mask the symptom):

1. **`Guid.NewGuid()` in `EntityManager.CreateEntity()`** — Every new entity got a random GUID, making telemetry entity IDs differ between runs. Fixed by using a per-instance counter (`_idCounter`) to generate sequential, reproducible GUIDs.

2. **`HashSet<Entity>` iteration order in `EntityManager._componentIndex`** — `Entity.GetHashCode()` defaulted to `RuntimeHelpers.GetHashCode()` (memory-address based), making `Query<T>()` iteration order non-deterministic across process runs. Fixed by overriding `Entity.GetHashCode()` to use `Id.GetHashCode()`. Contract is preserved: reference equality implies same Id implies same hash code.

3. **`System.Random` in `MovementSystem`** — Already seeded with 42, but the seed was hardcoded. Now uses `sim.Random` (the bootstrapper's `SeededRandom`) so the wander sequence is part of the unified seed chain.

4. **Wall-clock `capturedAt` in `AiReplayCommand`** — Uses `DateTimeOffset.UnixEpoch.AddSeconds(gameTime)` instead of `DateTimeOffset.UtcNow`, making `capturedAt` deterministic. (The determinism test strips `capturedAt` per the WP spec anyway — this is belt-and-suspenders.)

## Design decisions

**`InvocationContext.ExitCode` instead of `Environment.Exit()`** — All command handlers set `ctx.ExitCode` rather than calling `Environment.Exit()`. This allows `ECSCli.Tests` to call `AiCommand.Root.InvokeAsync(args)` in-process and read the returned exit code without terminating the test runner. `Environment.Exit(code)` is called exactly once, in `Program.cs`, after `InvokeAsync` returns.

**`System.CommandLine` scoped to `ai` subtree** — The existing bespoke parser in `CliOptions.Parse()` is completely untouched. The delegation guard `args.Length > 0 && args[0] == "ai"` routes before any existing code runs.
