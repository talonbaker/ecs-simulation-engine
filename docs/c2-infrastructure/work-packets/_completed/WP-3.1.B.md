## Architectural Decisions and Assumptions

### 1. Dual-source renderer (key assumption)

`NpcSilhouetteRenderer` reads from two sources:
- `EngineHost.WorldState` (WorldStateDto) — entity list and positions
- `EngineHost.Engine` (EntityManager) — `SilhouetteComponent`, `IntendedActionComponent`, `LifeStateComponent`, `MoodComponent`, `ChokingComponent`, `FacingComponent`

**Reason:** WorldStateDto schema v0.4.0 does not expose silhouette descriptors, intended-action kind, mood, or life state. Extending the DTO was out of scope for this packet (WP spec says "do not modify cast generator or SilhouetteDescriptorComponent"; the DTO is in Warden.Contracts and a version-bump there touches the WARDEN telemetry contract).

**Gap to close:** When WorldStateDto reaches v0.5.x and adds silhouette/animation fields, NpcSilhouetteRenderer should migrate to reading those fields from the DTO exclusively, restoring the clean single-source architecture.

### 2. SilhouetteComponent.Hair carries only hair length/style, not color

The `SilhouetteComponent.Hair` field stores a single string (e.g. "short", "medium") per the `CastSpawnComponents.cs` definition. The WP-3.1.B design notes reference both `hairLength` and `hairColor` as separate JSON fields. These are reconciled in `silhouette-catalog.json` as informational-only fields. At v0.1, hair sprites are keyed by the single `Hair` string and there is no per-NPC hair-color tinting.

**Path forward:** When a `HairColorComponent` is added to the ECS schema, `SilhouetteAssetCatalog.GetHairSprite()` should accept both length and color as separate parameters, and the catalog's `HairSpriteEntry` array should be expanded to `HairSpriteEntry.HairLength + HairColor → Sprite`.

### 3. Build value terminology: "slight" vs "slim"

The cast bible and `SilhouetteComponent` use "slight" for the lightest build. The WP-3.1.B design notes say "slim". Implementation uses "slight" to match the actual component schema. `silhouette-catalog.json` uses "slight". The `SilhouetteAssetCatalog.asset` serialized array also uses "slight".

### 4. Sprite art status: PLACEHOLDER

All 27 sprites in `ECSUnity/Assets/Sprites/Silhouettes/` are 32×64 solid-color PNGs generated by Python script. They are not pixel art. They are sufficient for:
- Instance-creation and hierarchy tests (AT-01)
- Position-tracking tests (AT-02)
- Tint-color tests (AT-03) — tint is independent of sprite content
- Batching material-sharing tests (AT-11)
- Performance gate (AT-10) — cost is per-SpriteRenderer, not per-pixel

**Final art:** Replace these files with authored pixel-art sprites from the art pipeline. No code changes required — the SilhouetteAssetCatalog.asset sprite reference array will be updated in the Editor to point to the new files.

### 5. Animator Controller: no Animation Clips at v0.1

The `NpcAnimator.controller` defines all 7 states (Idle, Walk, Sit, Talk, Panic, Sleep, Dead) and 6 bool parameters with transitions, but assigns no Animation Clips to the states. The Animator transitions between states when `NpcAnimatorController` sets the bool parameters.

At v0.1, "animation" means state-based sprite selection (handled in `NpcSilhouetteInstance.SetAnimationState`). Actual keyframed animation clips (breathing sway, walking cycle, gesture loop) are a future art-pipeline packet.

### 6. ChibiEmotionSlot tested only on the no-sprite path

The AT-08 tests exercise `Show(kind)` and `Hide()` without sprites loaded. This satisfies the packet requirement. The slot was NOT tested with a sample sprite loaded (the packet notes this is out of scope — WP-3.1.E ships the sprite content).

### 7. `silhouette-catalog.json` covers 10 archetypes

The cast bible lists 10 archetypes. The WP text says "13 archetypes (or however many are in the bible)". After counting all archetypes in `cast-bible.md`, there are exactly 10. The WP's mention of 13 appears to be an overcount from an earlier draft. All 10 cast-bible archetypes are present in the catalog with valid dominantColor hex values.

## Performance Analysis

### FPS gate (AT-10)

FPS measurements with silhouettes have NOT been run (cannot execute Unity from Cowork/shell). The performance analysis below is architectural reasoning:

**Per-NPC cost increase over NpcDotRenderer:**
- Dot renderer: 1 quad × 1 Material → 1 draw call (entire scene)
- Silhouette renderer: 4 SpriteRenderers × 30 NPCs = 120 SpriteRenderer instances

**Mitigations applied:**
1. **4 shared Materials** (one per layer) — all body SpriteRenderers share `_bodyMaterial`, etc.
2. **SpriteRenderer.color for tint** — vertex color, not material property; does not break dynamic batching
3. **No per-frame allocations** — entity map rebuilt only on entity count change; SpriteRenderer component instances are never re-created per frame
4. **Conditional state writes** — `NpcSilhouetteInstance.SetAnimationState` and `NpcAnimatorController.WriteAnimatorState` both guard on state change

**Expected draw call count at 30 NPCs (15 from office-starter, up to 30 if world file is expanded):**
- Body layer: 1 draw call (all share _bodyMaterial; vertex color difference is OK for dynamic batching when sprites share a texture atlas)
- Hair layer: 1 draw call
- Headwear layer: 1 draw call (invisible instances still contribute a batch slot; null sprites = empty mesh)
- Item layer: 1 draw call
- EmotionOverlay: 0 draw calls (all renderers disabled at v0.1)
- Room geometry: ~2 draw calls
- **Total: ~6 draw calls** — within the ≤10 spec

**IMPORTANT:** Full batching requires sprites from the same layer to share a Texture2D (Sprite Atlas). The placeholder sprites each have their own small texture, which WILL break batching in practice. At v0.1, with 27 individual small textures, the actual draw call count is likely higher (up to 4 per layer × n unique textures). When final art lands, pack all sprites into per-layer Sprite Atlases to restore batching to ~4 draw calls.

If the FPS gate fails due to draw call count, the fix is:
1. Create 4 Sprite Atlases (one per layer) in the Editor
2. Re-pack all sprites into the atlases
3. No code changes required

### Draw call count (AT-11)

The `SpriteBatchingTests.cs` test verifies:
1. Correct SpriteRenderer count (4 × NPC count)
2. All body renderers share the same `sharedMaterial` reference
3. Theoretical ceiling ≤ 10

Actual GPU draw count must be verified manually via Frame Debugger.

## Inspector Wiring Required (for the developer)

After opening the Unity project:

1. **SilhouetteAssetCatalog.asset** — open in Inspector and:
   - Assign `Sprites/Default` material (or custom Sprites/Default-based material) to all 4 material slots
   - Drag sprite files from `Sprites/Silhouettes/` into the appropriate sprite arrays (or leave null for placeholder behavior)

2. **NpcSilhouetteRenderer** on the scene's renderer GameObject — assign:
   - `_engineHost` → EngineHost
   - `_catalog` → Assets/Settings/SilhouetteAssetCatalog.asset
   - `_animatorController` → Assets/Animations/NpcAnimator.controller

3. **NpcDotRenderer** — disable or remove from the scene. It still compiles as a fallback but should not run alongside NpcSilhouetteRenderer on the same set of entities.
