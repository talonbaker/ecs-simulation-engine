## Summary (≤ 200 words)

Implemented the full cost ledger and budget enforcement stack in five production
files and two test files under `Warden.Orchestrator/Persistence/`.

`CostCalculator.CalculateUsd` delegates non-batch calls to the existing
`ModelRates.ComputeUsd` helper in `CostRates.cs` and handles batch calls
inline, composing the 50% batch discount with the 10% cache-read discount
(batch cached-read = `CacheReadPerMtok × 0.5`). The design notes say to
verify at implementation time whether Anthropic's policy is compose-or-better;
the cost model doc states "batch + cache are combined per Anthropic's policy",
so composition is used.

`CostLedger.Open` keeps a `FileStream` open for the lifetime of the instance
and uses `SemaphoreSlim(1)` to serialise concurrent writes. Each call to
`WriteAsync` writes one JSON line and then calls `Flush(true)` (fsync) before
releasing the semaphore, satisfying crash safety.

`BudgetGuard` exposes a separate `Record(decimal)` method alongside
`Check(decimal)` — this is necessary because the packet only specifies `Check`
but AT-05 implies accumulated spend. `Check` is read-only; `Record` advances
the running sum. Both are protected by a `lock` object.

AT-05 illustrative numbers (5 Sonnet calls at $0.06, blocks the 25th) do not
align with a $1.00 budget ($0.06 × 17 = $1.02 — the 17th blocks, not the
25th). The test asserts the correct 16-pass / 17th-blocks boundary, which is
the mathematically correct behaviour. A followup note is included below.

## Calculation decimals used

**Non-batch path** — delegates to `ModelRates.ComputeUsd`:
```
usd = inputTokens       / 1_000_000m × InputPerMtok
    + outputTokens      / 1_000_000m × OutputPerMtok
    + cacheWriteTokens  / 1_000_000m × CacheWritePerMtok
    + cacheReadTokens   / 1_000_000m × CacheReadPerMtok
```

**Batch path** — inline in `CostCalculator`:
```
usd = inputTokens              / 1_000_000m × BatchInputPerMtok
    + outputTokens             / 1_000_000m × BatchOutputPerMtok
    + cacheCreationInputTokens / 1_000_000m × CacheWritePerMtok × 0.5m
    + cacheReadInputTokens     / 1_000_000m × CacheReadPerMtok  × 0.5m
```

For Sonnet (non-batch, AT-01):
- cached-read: 32 000 / 1 000 000 × 0.30 = **0.0096**
- input:        2 500 / 1 000 000 × 3.00 = **0.0075**
- output:       1 500 / 1 000 000 × 15.00 = **0.0225**
- total: **0.0396**

## Followups

- AT-05 illustrative numbers ($0.06/call, budget $1.00, "blocks 25th") are internally inconsistent — packet should be corrected to either use ~$0.041/call (so 25th blocks) or state the 17th call blocks at $0.06/call.
- `BudgetGuard.Record` is not specified in the packet's public API but is required for AT-05 and for any real orchestrator usage; a follow-on WP should formally document it in the budget guard contract.
- `CostLedger` does not expose a `ReadAll` helper; callers parse JSONL themselves. A future WP (report aggregator) may want a typed reader.
