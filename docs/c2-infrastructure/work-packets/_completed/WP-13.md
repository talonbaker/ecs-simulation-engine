## Summary (≤ 200 words)

Wired `IWorktreeDiffSource` into `SonnetDispatcher.RunAsync` so that banned-pattern detection
becomes live policy rather than dead code.

Three supporting types (`IWorktreeDiffSource`, `GitWorktreeDiffSource`, `NullWorktreeDiffSource`)
already existed on the branch. This packet completed the remaining gaps:

1. `SonnetDispatcher` was updated to accept `IWorktreeDiffSource` as a constructor parameter.
   After schema validation and before returning, the dispatcher calls `GetDiffAsync`, evaluates
   the verdict through `FailClosedEscalator.Evaluate`, writes the ledger entry (the call was
   made; the money was spent), persists the CoT result, then — only if `result.Outcome == Ok`
   and `verdict.TerminalOutcome == Blocked` — overrides the return value to
   `outcome=blocked / reason=tool-error`.

2. `ResumeCommand` was updated to wire the diff source (it had the old 7-parameter constructor).
   `RunCommand` was already wired.

3. `SonnetDispatcherBannedPatternTests` was written covering all eight acceptance tests.

## Followups

- `GitWorktreeDiffSource` does not redirect stderr; a verbose git error to stderr can fill the stderr pipe buffer on very large error outputs. Low-priority given git diff error messages are typically short.
- The 1 ms timeout in AT-05 relies on process-startup latency being >1 ms. Works reliably on current hardware; a future machine could theoretically complete `git diff` in <1 ms on an empty repo if git is cached. Consider replacing with a process-executor abstraction for a fully hermetic test.
