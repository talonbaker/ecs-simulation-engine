## Summary (≤ 200 words)

Implemented the four-slab prompt model as specified. Four source files were added under `Warden.Orchestrator/Cache/`: the `CacheDisposition` enum, the `PromptSlab` record, `CachedPrefixSource` (corpus loader with mtime-based in-memory cache), and `PromptCacheManager` (request assembler).

`CachedPrefixSource` takes a manifest path and a base directory at construction. It validates every listed corpus file at construction time (AT-09 fail-fast). Corpus text is loaded lazily on the first `GetCorpusText()` call and cached; the cache invalidates when any corpus file's mtime or the manifest's own mtime changes (AT-04, AT-08).

One judgement call: the WP deliverables table says `CachedPrefixSource` "assembles slab 1" while the public surface section distinguishes role frame (slab 1) from corpus docs (slab 2). I followed the public surface section — the class exposes `GetRoleFrameText()` for slab 1 and `GetCorpusText()` for slab 2. AT-07 and AT-08 in the acceptance tests are written against this two-method model and pass cleanly.

A placeholder `docs/engine-fact-sheet.md` was created so the Phase-0 manifest entry does not fail AT-09's construction-time validation. The file notes it should be regenerated with `ECSCli ai describe`.
